CODE REVIEW: SizeShiftsImplicationsV2

PART 1 — PACKAGE DESIGN AND FUNCTIONALITY
================================================================================

Overall Structure
-----------------
The package is well-organized around R convention. Four public exports
(run_scenarios, make_figures, param_configs, convert_legacy_to_ssi) provide a
clean external API. Internal concerns (model engine, helpers, logging, config
building) are properly hidden.

Strengths:
  - CRN (common random numbers) reproducibility via withr::with_seed() with
    explicit offset seeds is well-implemented and documented
  - Parallel execution via future/future.apply with correct anti-BLAS
    over-subscription handling in .onLoad()
  - Comprehensive logging system (8 CSV logs, 7a-7h) with parallel-safe
    shard-then-merge strategy
  - ssi_run object structure provides clear provenance (timestamp, pkg_version,
    scenario_grid_hash)
  - convert_legacy_to_ssi is a thoughtful backward-compatibility bridge


Design Issues
-------------

1. %||% defined in two files.
   model_helpers.R:432 and log_helpers.R:7 both define %||%. R will load both,
   and which one "wins" depends on load order. Since both have identical
   semantics this doesn't cause errors, but it violates DRY. Move the
   definition to a single utils.R or zzz.R.

2. firstrev is hard-coded in two places.
   run_model.R:302 sets firstrev <- 20 locally, and
   run_scenarios_parallel.R:79 sets firstrev <- 20L with a comment
   "hard-coded in run_model()". If one is changed, the other won't match, and
   the saved parameters$firstrev in the ssi_run will no longer reflect what
   the model actually used. It should come from the profile config.

3. No tests.
   There is no tests/testthat/ directory and no test files. For a simulation
   with stochastic elements, even a handful of smoke tests would catch
   regressions when the model is modified (e.g., run one iteration, check
   output structure; check CRN reproducibility; check validate_config catches
   bad inputs).

4. ages = 1:9 is an implicit global constraint.
   The ages parameter is passed through configs, but several parts of the model
   assume ages are consecutive integers starting at 1 (see Part 2, Issue 5).
   If someone tried ages = 3:8 (a biologically reasonable change), multiple
   things would fail silently.

5. DESCRIPTION imports are broader than needed.
   readxl, tidyselect, and forcats are in Imports: but may only appear in
   figure files. Unused Imports: inflate the package's dependency footprint.

BUG 1 (CRITICAL): Regime shift sets alpha to the OLD regime, not the new one
------------------------------------------------------------------------------
Location: run_model.R:344-350

  if (reglength != 0) {
    if (withr::with_seed(seednum + 6000L + y,
                         sample(c(rep(0, reglength - 1), 1L), 1L))) {
      if (aset == 1) alpha <- alpha.low else alpha <- alpha.high   # <-- wrong
      aset <- abs(aset - 3) ## change regime indicator
    }
  }

When a regime shift triggers, alpha is set based on aset BEFORE it flips.
aset then flips with abs(aset - 3). The result:

  - Shift from low->high (aset=1): alpha set to alpha.low (stays low!),
    aset flips to 2 (high)
  - Shift from high->low (aset=2): alpha set to alpha.high (stays high!),
    aset flips to 1 (low)

Alpha is always re-affirmed to the old regime. The state indicator changes,
but the productivity parameter does not change until the NEXT shift event.
Every regime change is delayed by one shift cycle.

Fix:
  aset <- abs(aset - 3)                     # flip first
  alpha <- c(alpha.low, alpha.high)[aset]   # then assign to NEW regime


BUG 2 (SIGNIFICANT): Initial-year fecundity/eggmass uses male age composition
           for female sizes
------------------------------------------------------------------------------
Location: run_model.R:233-252

  age_comp_y <- .calc_agecomp(
    ages = ages, recruits = round(PopDat$Esc[y] * (1 - propfemale[y])),  # male count
    meanage = meanage[y] - agediff / 2, sdage = sdage,                   # male mean age
    seed = seednum + 515L + y
  )
  sizes_y <- rep(meanSaA[y,], prop.table(age_comp_y) * PopDat$Esc[y])   # scaled to ALL fish

For fecundity/eggmass calculation, sizes_y should represent female fish. But
the age composition is drawn using male headcount and the male mean age
(meanage - agediff/2). Those proportions are then scaled to TOTAL escapement
(both sexes). The resulting sizes_y is inconsistent -- it has male age
structure but is scaled to total fish count.

Compare to the main simulation loop (lines 460-468, section 7), which
correctly builds sizes_y from female escapement by age. The initialization
period (y in 1:nyi) uses the wrong sex.

This affects only the nyi=10 spinup years, but the recruits generated during
spinup seed the age structure of early simulation years.


BUG 3 (SIGNIFICANT): k (iteration counter) overwritten by age loop variable
------------------------------------------------------------------------------
Location: run_model.R:16 and run_model.R:284, 512-516

  iter <- config$k   # line 16: iteration number stored in 'iter'
  ...
  for (k in 1:nage) {           # lines 284, 512: k reused as age index
    ret_by_age_sex[y + k, k, 1] <- ...
  }

The unpacked iteration number is stored as 'iter', but 'k' is also used as
the age loop variable. After the age loop, k = nage, not the original
config$k value. This doesn't cause a functional problem currently because
config$k isn't used directly after unpacking -- but it's a trap for future
code modifications.


ISSUE 4 (POTENTIAL ERROR): Lognormal bias correction drops AR(1) variance
           inflation term
------------------------------------------------------------------------------
Location: run_model.R:569, 664, 710, 757

  alpha_sim <- exp(log_a_sim + 0.5 * sig_sim^2)
  # Comment: "Recoded to drop phi from the lognormal bias correction"

For a GLS with AR(1) errors, the correct mean-to-median adjustment is:
  0.5 * sigma^2 / (1 - phi^2)

The code uses only:
  0.5 * sigma^2

This is smaller than the theoretically correct value when phi > 0. Both
parameter profiles have positive rho (0.4 for Ohlberger, 0.593 for
Kuskokwim), and the GLS-estimated phi will also be positive. This means alpha
estimates are systematically lower than they should be.

The code comment says this was "recoded" -- so this may intentionally match
the published Ohlberger et al. (2025) code, or it may reflect a correction to
a different error. Either way, the formula diverges from the theoretical
lognormal bias correction and should be verified against the paper's methods
section.


ISSUE 5 (MINOR): Harvest-by-age indexing silently assumes ages starts at 1
------------------------------------------------------------------------------
Location: run_model.R:434-437

  harv_prop_M <- ret_prop_by_sex_y[as.numeric(names(tab)), 1]
  harv_by_age_sex[y, as.numeric(names(tab)), 1] <- round(tab * fac * harv_prop_M)

names(tab) are the sampled age VALUES (e.g., "4", "5", "6"). These are used
directly as row INDICES into ret_prop_by_sex_y (which is indexed 1:nage).
This works only because ages = 1:9, making age values equal to their positions.

If ages were changed to c(3, 4, 5, 6, 7) or any non-1-starting sequence,
the indexing would be wrong or cause out-of-bounds errors with no warning.


ISSUE 6 (MINOR): DLM AICc uses incorrect parameter count
------------------------------------------------------------------------------
Location: model_helpers.R:71, 86, 101

  npara <- 3 + sum(c(var_alpha, var_beta))
  dlm_out <- dlm::dlmMLE(y = lnRS, build = build_mod, parm = c(-0.1, -0.1, -0.1), ...)
  AICc <- 2 * lls + 2 * npara + (2 * npara * (npara + 1) / (length(data$Rec) - npara - 1))

The starting parm vector always has 3 elements, so dlmMLE always optimizes
over 3 parameters. But npara is 4 (one time-varying param) or 5 (both), which
inflates the AICc penalty by 2.

AICc is returned but not used for management decisions in run_model.R, so
this doesn't affect simulation outcomes -- but the returned AICc values are
not meaningful for model comparison.


ISSUE 7 (MINOR): Inconsistency in future trend freeze
------------------------------------------------------------------------------
Location: run_model.R:163, 174, 195

When futureT == "no":
  - meanage is frozen at the LAST SINGLE VALUE of the historical period
    (meanage[max_yh])
  - propF_y and meanSaAdet are frozen at the 5-YEAR MEAN of the end of
    the historical period

This inconsistency is minor in practice but could matter if historical
trajectories are highly variable near their end.

================================================================================
PRIORITY NOTES
================================================================================

Bug 1 (regime shift) is the most likely source of incorrect simulation
results. The productivity regime never changes in the same year as the shift
event; it's always delayed. With reglength = 20 (~5% chance of shift per
year), this could materially affect outcomes in scenarios with regime shifts
enabled (regstr > 1).

Bug 2 affects only the nyi = 10 initialization period but means the early
spinup generates recruits from fecundity computed with the wrong sex structure.

Issue 4 (lognormal bias correction) should be verified against Ohlberger et
al. (2025) before deciding whether it's a bug or an intentional simplification.
